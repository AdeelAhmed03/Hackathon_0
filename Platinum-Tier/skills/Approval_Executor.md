# Approval_Executor

## Extended Skill Definition

### Overview
Approval_Executor processes approval requests and executes approved actions using local MCP servers and credentials. This skill bridges the gap between cloud-generated drafts and local execution, ensuring all external actions go through proper human approval and use local credentials.

### Detailed Process Flow

#### Step 1: Approval Validation
- Verify the approval request file is properly moved to `/Approved/`
- Check YAML frontmatter for approval status and parameters
- Validate that the action was previously generated by cloud agent
- Confirm the action type is supported for execution

#### Step 2: Action Type Determination
- Parse action type from YAML frontmatter (`type: email_send`, `social_post`, `payment_execute`, etc.)
- Extract action-specific parameters and content
- Validate that all required fields are present
- Check against Platinum Tier execution policies

#### Step 3: MCP Server Selection
Based on action type, select appropriate MCP server:
- **Email actions**: Use `email-mcp` server
- **Social media actions**: Use `social-mcp` server
- **Accounting/Financial actions**: Use `odoo-mcp` server
- **Other actions**: Route to appropriate specialized MCP

#### Step 4: Local Credential Injection
- Retrieve local credentials from secure storage
- Inject credentials into MCP server call
- Ensure cloud credentials are never used for execution
- Validate credential availability before proceeding

#### Step 5: Action Execution
- Make JSON-RPC call to selected MCP server with action parameters
- Include local credentials and execution context
- Wait for response from MCP server
- Handle MCP server errors appropriately

#### Step 6: Response Processing
- Parse MCP server response (success/failure, error details, execution data)
- Update execution status in approval request
- Extract relevant execution metrics (ID, timestamp, etc.)
- Prepare response for dashboard update

#### Step 7: Dashboard Update
- Update execution metrics in Dashboard.md
- Record execution timestamp and result
- Update success/failure counters
- Generate dashboard update file for cloud sync

#### Step 8: Completion Handling
- Move completed task to `/Done/local/`
- Log execution in audit logs with full details
- Generate completion notification if applicable
- Clean up temporary execution data

### Action Types Supported

#### Email Execution
- **Action**: `email_send`, `email_reply`, `email_forward`
- **Process**: Use local SMTP credentials to send actual email
- **MCP**: `email-mcp` server
- **Output**: Email ID, send status, delivery confirmation

#### Social Media Execution
- **Action**: `facebook_post`, `instagram_post`, `twitter_post`
- **Process**: Use local API credentials to post to platforms
- **MCP**: `social-mcp` server
- **Output**: Post ID, platform confirmation, engagement metrics

#### Financial Execution
- **Action**: `payment_execute`, `invoice_send`, `transaction_record`
- **Process**: Use local Odoo credentials to execute financial actions
- **MCP**: `odoo-mcp` server
- **Output**: Transaction ID, financial confirmation, accounting updates

### Examples

#### Example Email Execution
**Input Approval File** (`Approved/APPROVAL_email_001.md`):
```yaml
---
type: approval_request
action: email_send
original_task: CLOUD_EMAIL_DRAFT_001
approver: local_user
approved: true
---
## Email Approval Request

**To**: customer@example.com
**Subject**: Order Confirmation #12345
**Body**: Thank you for your order. Your items will ship tomorrow.
```

**Processing**:
1. Validate the approval and extract email details
2. Select `email-mcp` server
3. Inject local SMTP credentials
4. Execute `send_email` method with parameters
5. Receive response with email ID and send status

**Output**:
- Email sent successfully with ID: `msg_1234567890`
- Dashboard updated with send metrics
- Task moved to `/Done/local/APPROVED_email_001.md`

#### Example Social Media Execution
**Input Approval File** (`Approved/APPROVAL_social_002.md`):
```yaml

type: approval_request
```

- Task moved to `/Done/local/APPROVED_social_002.md`

### Error Handling Scenarios

#### MCP Server Unavailable
- **Issue**: Selected MCP server is not responding
- **Action**: Queue for retry with exponential backoff (2s, 4s, 8s)
- **Log**: Record connection failure for monitoring
- **Retry**: Up to 3 attempts before marking as failed

#### Credential Issues
- **Issue**: Local credentials are invalid or expired
- **Action**: Stop execution, alert user to update credentials
- **Log**: Record credential failure in security logs
- **Status**: Mark as failed with credential error

#### Action Validation Failure
- **Issue**: Required action parameters are missing or invalid
- **Action**: Don't execute, mark as failed with validation error
- **Log**: Record validation failure for debugging
- **Status**: Move to failed task queue

#### Platform-Specific Errors
- **Issue**: Platform rejects the action (rate limit, content policy, etc.)
- **Action**: Analyze error type and respond appropriately
- **Log**: Record platform-specific error details
- **Retry**: For rate limits, wait and retry; for policy violations, mark failed

### Security Considerations
- **Local Credentials Only**: Never use cloud credentials for execution
- **Execution Validation**: Verify all parameters before execution
- **Approval Requirement**: Execute only approved actions
- **Audit Trail**: Complete logging of all execution activities
- **Error Sanitization**: Don't expose credential details in error messages

### Integration Points
- **Email MCP**: For email sends using local SMTP credentials
- **Social MCP**: For social media posts using local platform credentials
- **Odoo MCP**: For accounting actions using local Odoo credentials
- **Dashboard**: Updates with execution metrics
- **Audit Logger**: Full execution logging
- **File System**: Task completion and status updates
